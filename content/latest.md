# 최신

## 최신 사업별 감리 점검 포인트

1. 클라우드 네이티브 전환 사업
단순히 서버를 옮기는(Lift & Shift) 것이 아니라, 클라우드의 장점을 극대화하는 구조로 전환하는 것이 핵심입니다.

- **MSA(마이크로서비스 아키텍처) 적정성**: 서비스 간 결합도(Coupling)를 낮추고 독립적인 배포가 가능한 구조인지 점검.
- **컨테이너 및 오케스트레이션**: Docker/Kubernetes 도입 시 자원 할당(Resource Quota) 및 오토스케일링(Auto-scaling) 설정의 적정성.
- **CI/CD 파이프라인**: 빌드-테스트-배포 자동화 과정에서 보안 취약점 점검(DevSecOps)이 포함되었는지 확인.
- **클라우드 네이티브 보안**: 컨테이너 이미지 보안 스캔, 런타임 보안 모니터링 체계 구축 여부.

2. 인공지능(AI) 및 초거대 AI 도입 사업
최신 **'초거대 AI 도입·활용 가이드라인 2.0 (2025.04)'**과 'AI 윤리기준 자율점검표(2025.02)' 준수가 핵심입니다.

- **데이터 거버넌스**: 학습 데이터의 저작권 준수, 개인정보 비식별화 처리의 적정성 점검.
- **할루시네이션(환각현상) 방지**: RAG(검색 증강 생성) 기술 적용 여부 및 답변의 정확도 검증 체계.
- **보안 및 프라이버시**: 프롬프트 주입(Prompt Injection) 공격 방어 및 사용자 입력 데이터의 외부 유출 방지 대책.
- **AI 윤리 및 투명성**: 결과값에 대한 설명 가능성(Explainability)과 편향성 방지를 위한 필터링 정책 확인.

3. 대규모 DB 통합 및 이관 사업
'공공기관의 데이터베이스 표준화 지침 (2024.11)' 개정안에 따른 데이터 품질 관리가 핵심입니다.

- **데이터 표준 준수**: 범정부 데이터 표준 단어·용어·도메인 준수 및 기관 표준사전 정립 여부.
- **이관(Migration) 정합성**: 구 시스템에서 신 시스템으로 데이터 이관 시 레코드 수 일치 및 데이터 변환 로직의 정확성 검증.
- **데이터 품질 관리**: 값의 유효성, 비즈니스 규칙 준수 등 품질 진단 도구 활용 및 결과 조치 확인.
- **성능 최적화**: 대량 데이터 처리를 위한 인덱스 설계, 쿼리 최적화 및 파티셔닝 전략의 적정성.

4. 민간 클라우드(SaaS/PaaS) 이용 사업
공공기관이 민간 서비스를 안전하게 이용하기 위한 **보안인증(CSAP)**과 서비스 수준(SLA) 확인이 핵심입니다.

- **CSAP 인증 확인**: 도입하려는 SaaS/PaaS가 보안 등급(상·중·하)에 맞는 CSAP 인증을 보유하고 유효한지 확인.
- **SLA(서비스 수준 협약)**: 가용성(예: 99.9% 이상), 장애 보상 범위, 데이터 백업 및 복구 조건이 구체적인지 점검.
- **데이터 주권 및 파기**: 계약 종료 시 데이터 반환 방법과 완전 파기 확인서 제출 등 사후 관리 절차 수립 여부.
- **망 분리 및 접근 제어**: 공공기관 내부망과 민간 클라우드 간의 연계 시 보안 게이트웨이 및 암호화 적용 여부.

각 사업 유형별로 감리 현장에서 즉시 활용할 수 있는 **핵심 점검 항목(Checklist)**을 정리해 드립니다. 이 내용은 최신 고시와 가이드라인을 기반으로 실무적인 관점에서 구성되었습니다.

## 기술적 점검 방법 및 결과 기록 양식

살펴본 4가지 주요 사업 유형에 대해 감리 현장에서 실제 사용하는 **기술적 점검 방법(How-to)**과 결과 기록 양식(Template) 예시를 정리합니다.

### 1. 클라우드 네이티브: 기술 점검 및 양식

<details>
<summary>펼치기</summary>

클라우드 환경은 설정(Configuration)이 곧 인프라입니다. 따라서 코드화된 인프라(IaC) 및 설정값 검증이 핵심입니다.

**기술적 점검 방법:**
- **Auto-scaling 검증**: 부하 발생 도구(JMeter 등)를 사용하여 설정된 임계치(CPU 70% 등)에서 Pod가 자동으로 늘어나는지 모니터링 툴(Grafana, Lens 등)로 확인.
- **장애 격리 검증**: 특정 마이크로서비스를 강제로 중단시켰을 때, API Gateway에서 해당 서비스로의 요청을 차단(Circuit Breaking)하고 타 서비스는 정상 작동하는지 확인.

**점검 양식(예시):**

| 점검항목 | 테스트 시나리오 | 기대 결과 | 실제 결과 | 판정 |
|----------|----------------|-----------|-----------|------|
| 자동 확장성 | CPU 부하 80% 이상 지속 발생 | Replica Set이 2개에서 5개로 증가 | 5개로 정상 증설 확인 | 적합 |
| 장애 전파방지 | 결제 서비스 강제 종료 | 메인 페이지는 정상 호출됨 | 타 서비스 영향 없음 확인 | 적합 |

</details>

### 2. 인공지능(AI): 기술 점검 및 양식

<details>
<summary>펼치기</summary>

AI 감리는 모델의 **성능 지표(Metric)**와 데이터 흐름을 기술적으로 검증해야 합니다.

**기술적 점검 방법:**
- **성능 평가서 검증**: 수행사가 제출한 Test Set 기반의 혼동 행렬(Confusion Matrix)을 재검증. (Precision, Recall, F1-Score 등 계산)
- **할루시네이션 테스트**: 시스템 프롬프트(System Prompt) 외의 질문이나 유도 심문에 모델이 거부하거나 정확한 근거(RAG)를 대는지 직접 질의응답(QA) 수행.

**점검 양식(예시):**

| 평가 지표 | 목표치 | 측정 결과 | 검증 도구 | 비고 |
|-----------|--------|-----------|-----------|------|
| 정확도(Accuracy) | 92.0% 이상 | 94.5% | Python Scikit-learn | 테스트 데이터셋 1만 건 |
| 응답 속도(Latency) | 2초 이내 | 1.8초 | API 호출 로그 분석 | GPU 서버 점유율 60% 시 |

</details>

### 3. DB 통합 및 이관: 기술 점검 및 양식

<details>
<summary>펼치기</summary>

데이터 이관은 샘플링이 아닌 전수 검증 원칙을 기술적으로 확인해야 합니다.

**기술적 점검 방법:**
- **데이터 무결성 검증**: SQL 쿼리를 통해 AS-IS와 TO-BE의 COUNT(*), SUM(금액), AVG(값) 등을 비교하여 데이터 유실 여부 확인.
- **표준 준수 자동 검증**: 공공 데이터 품질관리 도구(DQC) 등을 활용하여 표준 단어/용어 사전과 실제 스키마의 일치도를 자동 점검.

**점검 양식(예시):**

| 테이블명 | AS-IS 건수 | TO-BE 건수 | 오차 건수 | 주요 정합성 검증 쿼리 결과 |
|----------|------------|------------|-----------|---------------------------|
| 회원정보 | 1,200,500 | 1,200,500 | 0 | SELECT SUM(mileage) FROM users 일치 |
| 결제내역 | 5,430,210 | 5,430,210 | 0 | SELECT COUNT(*) FROM payment 일치 |

</details>

### 4. 민간 클라우드(SaaS): 기술 점검 및 양식

<details>
<summary>펼치기</summary>

민간 서비스는 보안 설정과 접속 로그의 가시성을 확보하는 것이 중요합니다.

**기술적 점검 방법:**
- **인증 보안성 테스트**: 관리자 페이지 접속 시 2차 인증(MFA) 요구 여부 확인 및 비밀번호 복잡도 정책 적용 확인.
- **데이터 파기 기술 확인**: 암호화된 데이터의 복구 불가능한 파기(Key Shredding) 알고리즘이 설계서 및 실제 가동 환경에 적용되었는지 확인.

**점검 양식(예시):**

| 보안 요구사항 | 설계 반영 여부 | 실제 구현 확인 | 확인 방법 | 판정 |
|---------------|----------------|----------------|-----------|------|
| MFA 적용 | Y | OTP/문자 인증 연동 | 실제 관리자 로그인 테스트 | 적합 |
| 비인가 차단 | Y | 지정된 IP 외 접근 차단 | 외부망 접속 시도(Access Denied) | 적합 |

</details>

## 기타 기술적 점검 방법

감리 대상 시스템의 기술적 검증을 보완하기 위해 아래의 추가적인 기술적 점검 방법을 참고하시기 바랍니다.

### 1. 클라우드 네이티브 (쿠버네티스 중심)

<details>
<summary>펼치기</summary>

인프라의 상태를 명령어로 직접 확인하여 설계서와 일치하는지 검증합니다.

**오토스케일링(HPA) 설정 확인**
- **명령어**: `kubectl get hpa -n [네임스페이스]`
- **점검 내용**: 최소/최대 Pod 수, CPU/Memory 임계치 설정이 설계와 일치하는지 확인.

**리소스 할당(Quota) 제한 확인**
- **명령어**: `kubectl describe quota -n [네임스페이스]`
- **점검 내용**: 특정 서비스가 전체 자원을 독점하지 않도록 limits와 requests가 설정되었는지 확인.

**서비스 메쉬(Istio 등) 서킷 브레이커 확인**
- **방법**: `istioctl proxy-config endpoint [포드명]` 명령으로 비정상 서비스 차단 설정(Outlier Detection) 확인.

</details>

### 2. 인공지능 (AI 모델 성능 및 자원)

<details>
<summary>펼치기</summary>

모델의 실제 구동 상태와 결과의 신뢰도를 기술적으로 검증합니다.

**GPU 점유율 및 가용량 확인**
- **명령어**: `nvidia-smi`
- **점검 내용**: 추론(Inference) 시 GPU 메모리 누수가 없는지, 메모리 점유율이 적정한지 확인.

**API 응답 로그 분석**
- **방법**: ELK 스택이나 클라우드 로그(CloudWatch 등)에서 응답 시간(Latency) 쿼리 수행.
- **쿼리 예시**: `filter @message like /request_time/ | stats avg(request_time) by bin(1m)`

**RAG(검색증강생성) 참조 출처 확인**
- **방법**: AI 답변 시 하단에 실제 DB/문서의 근거(Source Citation)가 링크되거나 로그에 남는지 확인.

</details>

### 3. DB 통합 및 이관 (SQL 검증)

<details>
<summary>펼치기</summary>

데이터 유실을 막기 위해 'Count'와 'Hash' 값을 비교하는 것이 정석입니다.

**데이터 정합성 전수 검정 (CheckSum)**
- **SQL 예시**:
  ```sql
  -- Oracle/PostgreSQL 예시
  SELECT COUNT(*), SUM(PAY_AMT), BIT_XOR(USER_ID) FROM TABLE_NAME;
  ```
- **점검 내용**: 원천(AS-IS)과 타겟(TO-BE) DB에서 동일 쿼리를 실행하여 값이 1원이라도 틀린지 확인.

**인덱스 및 실행계획(Explain Plan) 확인**
- **명령어**: `EXPLAIN ANALYZE SELECT ...`
- **점검 내용**: 이관 후 Full Table Scan이 발생하는지, 인덱스가 적절히 생성되어 성능을 보장하는지 확인.

</details>

### 4. 민간 클라우드/SaaS (보안 및 연계)

<details>
<summary>펼치기</summary>

외부망 연계 시 보안 설정과 데이터 암호화를 기술적으로 점검합니다.

**TLS 암호화 통신 확인**
- **명령어**: `openssl s_client -connect [도메인]:443`
- **점검 내용**: 최신 보안 프로콜인 TLS 1.2 또는 1.3만 허용되는지, 취약한 암호화 알고리즘은 차단되었는지 확인.

**API 인증 토큰 보안**
- **방법**: 호출 시 JWT(JSON Web Token) 등 인증 토큰의 만료 시간(Exp)이 짧게 설정되어 있는지 디코딩하여 확인.

**망 분리 접속 로그**
- **방법**: 클라우드 콘솔의 Flow Log 또는 CloudTrail을 조회하여 비인가 IP의 접근 시도가 차단(Deny)되었는지 로그 확인.

</details>
